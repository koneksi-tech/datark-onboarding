1. Platform & Infrastructure DONE

A shared managed-Kubernetes cluster + a dedicated Vault, ready to host many isolated tenants.

  • Managed Kubernetes cluster datark-cluster (NHN NKS, 4× m2.c16m32 = 64 vCPU / 128 GB, K8s v1.34.3)
  • Dedicated Vault node (125.6.39.130) — transit / AppRole / kv-v2, auto-unseal
  • Dynamic per-tenant block storage — Cinder CSI nhn-block-ssd (expandable)
  • Public entry — ingress-nginx + NHN LoadBalancer 133.186.214.160, wildcard TLS *.koneksi.co.kr
  • Image registry wired — NHN NCR (koneksi-tech) with per-namespace pull secrets

2. Automated Per-Tenant Onboarding DONE

One action provisions a fully isolated DatArk stack for a client — namespace-per-tenant, tier-sized.

Full stack, per tenant

Six cooperating services deployed into one namespace:

  • kripfs — decentralized storage (1–5 peered nodes)
  • backend — application API (only public surface)
  • redis, mongodb, postgres (TimescaleDB)
  • kripfs-db — DB→kripfs backup daemon

Isolation & secrets

Every tenant gets its own:

  • Kubernetes namespace + NetworkPolicy (default-deny) + ResourceQuota
  • Vault path, transit key & AppRole
  • kripfs bearer, DB passwords, wildcard TLS
  • bootstrapped admin account for the developer

Three ways to onboard

  • Console Onboard button (name + tier)
  • GitHub Actions provisioning workflow
  • datark-provision.sh (operator CLI)

Tiers & data-safe upgrade

free / starter / pro / enterprise. Upgrade grows kripfs (nodes + storage via online volume expansion) with zero data loss; databases stay fixed. Verified free→pro live.
