DatArk Onboarding — Status Update

Per-tenant provisioning of the DatArk stack on NHN. As of 2026-07-08. Environment is separate from prod Koneksi (own K8s cluster + own Vault).

🎯 Goal of this onboarding

Make DatArk sellable as a self-service product on a marketplace. When a client subscribes and picks a plan, the platform should automatically provision an isolated, private DatArk environment for that client — no manual setup.

Each client (tenant) gets their own dedicated stack on a shared Kubernetes cluster, isolated in its own namespace:

  • kripfs storage nodes (count scales with the plan) + backend API + redis + mongodb + postgres
  • reached at the client's own subdomain <client>.koneksi.co.kr
  • secrets isolated per-client in Vault; databases and storage private to that tenant

Provisioning is driven by one automated command (Helm + bash) that the marketplace "buy" button can call. Plans are tiered (free → starter → pro → enterprise), and a client can upgrade to get more kripfs nodes / replication on demand.

This status report shows how far that goal is: the prerequisites (K8s + Vault) are ready, all five components (incl. kripfs) now deploy and run in pods, and a tenant is reachable over valid TLS. Remaining: DNS records + a few hardening items.

4. Upgrade & scaling — client tier changes LIVE-VERIFIED

When a client upgrades their plan, the same provisioning command re-runs with a higher tier. It scales up kripfs pods + replication and resources — with no data loss.

The switch — one idempotent command: 
./bin/datark-provision.sh --tenant <client> --tier pro     # free → pro, etc.
Detected automatically as an upgrade (not a fresh install). Re-run any time to move up a tier.

Tier ladder — what scales

Tierkripfs podsReplication factorStorage / kripfs podNamespace quota (CPU)

Upgrading free → pro adds 2 more kripfs pods (1 → 3) and raises replication (1 → 2). pro → enterprise adds 2 more (3 → 5), replication → 3.

Data safety — verified on a live upgrade

GuaranteeStatusEvidence
Databases keep their data on upgradeVERIFIEDfree→starter run: mongo & postgres PVC UUIDs unchanged before/after
kripfs identity preserved on scale-upBY DESIGNeach pod's identity lives on its own retained PVC (StatefulSet volumeClaimTemplates)
New peers auto-joinBY DESIGNreplica count bump re-templates every pod's cluster_peers; rolling update reconciles the mesh
No downtime (3+ nodes)BY DESIGNStatefulSet rolls one pod at a time; quota pre-sized per tier
Caveats:
  • Storage grows, not shrinks. StorageClass allows volume expansion, but the StatefulSet template's PVC size is immutable on existing pods — growing kripfs/mongo/postgres disk is a manual PVC-expand step, not automatic on tier change.
  • Upgrade only (scale up) is the supported path; downgrades (fewer pods / smaller disk) are not automated.
  • kripfs pod scaling is designed & renders, but end-to-end scale-up is unverified until the kripfs image exists (§3).

6. Next task — self-service provisioning via GitHub Actions TO DO

Goal: a manually-triggered GitHub Action in datark-onboarding that provisions a tenant on demand — the operator/marketplace "provision" button.

How it works

  • Trigger: workflow_dispatch (manual "Run workflow" button) — requires input before it runs.
  • Inputs:
    • client name (required) → used as the tenant id and the namespace (tenant-<client>) where all DatArk resources deploy.
    • tier (dropdown: tier 1 / tier 2 / tier 3) → maps to a plan (e.g. starter / pro / enterprise).
  • Action: writes kubeconfig from secret → installs helm/kubectl/vault → runs ./bin/datark-provision.sh --tenant <client> --tier <tier> → outputs the tenant URL.
