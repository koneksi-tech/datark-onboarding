DatArk Marketplace Provisioning

On-demand, per-tenant provisioning of an isolated DatArk stack — kripfs (pods) + backend + redis + mongodb + postgres — onto a shared Kubernetes cluster. A bash entrypoint drives a single Helm chart; the marketplace "buy" button calls that script later.

✅ This design is now IMPLEMENTED and live-tested. All 5 components (incl. kripfs image kipfs:v0.0.1) run in pods across all tiers; provisioning is automated via a GitHub Actions workflow; tenants are reachable over valid TLS at <client>.koneksi.co.kr. This page is the original design record — for current state see STATUS.html, SPECIFICATIONS.html, and PROVISIONING-RUNBOOK.html. (Tenant domain is koneksi.co.kr, not the earlier datark.koneksi.co.kr placeholder.)
TL;DR
  • Model: one shared cluster, namespace per tenant. Each purchase = one namespace holding that tenant's full stack.
  • Mechanism: a bash wrapper (datark-provision.sh) calls helm upgrade --install on a single datark-tenant chart. Tiers are Helm values files.
  • Input: only --tenant <id> + --tier <starter|pro|enterprise>, plus a kube-admin context. Nothing else.
  • Access: per-tenant subdomain <id>.koneksi.co.kr via shared ingress-nginx + wildcard TLS. kripfs/redis/mongo stay internal.
  • Teardown: datark-deprovision.sh <id> → helm uninstall + delete namespace (and PVCs). Clean and total.
  • Blockers to clear first: kripfs pod image, per-tenant backend manifests, and a default StorageClass (hard blocker — stateful pods can't run without it).

4. How kripfs peers inside Kubernetes

The old Ansible plan flagged two hazards for dockerized kripfs: (a) identity must persist or a restart re-keys the node and breaks trust; (b) each node must announce an address peers can actually reach. A StatefulSet + headless Service solves both natively:

  • Stable identity → each replica gets its own PVC via volumeClaimTemplates; the kripfs identity lives there and survives pod reschedules. Gotcha (a) solved.
  • Stable, reachable address → headless Service gives every pod deterministic DNS: kripfs-0.kripfs-hl.tenant-abc.svc.cluster.local. That's the announce address and the peer address. Gotcha (b) solved.
  • Auto-generated peer list → a Helm range over the replica count templates cluster_peers (every ordinal except self) into a ConfigMap. Add a node = bump the replica count and re-run; peers regenerate.
  • Two shared secrets, same on every pod → cluster_secret (peer trust) and koneksi_static_bearer (the API bearer). Both delivered identically to all replicas from the per-tenant Vault-sourced Secret. There is no separate block key in the DatArk kipfs config.
cluster_peers (rendered for kripfs-1 in a 3-node tenant):
  http://kripfs-0.kripfs-hl.tenant-abc.svc.cluster.local:8080
  http://kripfs-2.kripfs-hl.tenant-abc.svc.cluster.local:8080

5. Provisioning process flow

datark-provision.sh --tenant abc --tier pro  ·  idempotent (safe to re-run)  ·  set -euo pipefail + rollback trap.

Input & validate

Parse --tenant --tier. Validate tenant-id (^[a-z0-9-]{3,30}$), tier exists, kube context reachable, tier values file present.

▼

Pre-flight checks

Confirm cluster reachable (kubectl auth can-i), StorageClass exists, ingress-nginx present, NCR pull secret available. Fail fast with a clear message if a blocker is missing.

▼

Idempotency guard

If namespace tenant-abc already has a Helm release → treat as upgrade (e.g. tier change), not a fresh create. Otherwise create-new.

▼

Provision Vault (per-tenant)

Using a provisioner token/AppRole, the script creates the tenant's Vault objects once (idempotent): KV secrets at secret/datark/tenants/abc/*, a per-tenant transit key transit/keys/datark-abc, a policy, and an AppRole → yields role_id/secret_id. Full detail in §8.

▼

Materialize the K8s Secret

Read the tenant's secrets from Vault and write the per-tenant K8s Secret (backend + kripfs + mongo + redis env), including the AppRole role_id/secret_id the backend uses to reach Vault transit at runtime. Generated once, reused on upgrades so identity/data stay valid. See §8/§7b.

▼

helm upgrade --install

helm upgrade --install tenant-abc ./datark-tenant -n tenant-abc --create-namespace -f tiers/pro.yaml --set tenant.id=abc --wait --timeout 10m. Renders kripfs StatefulSet, backend+ingress, redis, mongo, secret refs, policies.

▼

Wait for readiness

kubectl rollout status for each workload; --wait blocks until pods are Ready. kripfs pods come up ordinally and form their mesh.

▼

Verify (definition of done)

kripfs /health = 200 on each pod; each pod's cluster_peers lists the right peers; a test upload to one node is retrievable from another (replication proof); backend answers through its Ingress.

▼

Output access details

Print/return JSON: url=https://abc.koneksi.co.kr, admin bearer, tier, node count. This is what the marketplace hands the client.

▼

On any failure → rollback

Trap: helm rollback (upgrade) or helm uninstall + kubectl delete ns (fresh create) so a half-provisioned tenant never lingers. Exit non-zero with the failing step.

6. Deprovisioning flow

Input & confirm

datark-deprovision.sh --tenant abc. Require explicit confirm (or --force for the marketplace path). Verify the namespace/release exists.

▼

Optional data grace

If configured, snapshot/export mongo + kripfs blocks to object storage before deletion (retention window on cancellation).

▼

Uninstall & delete (K8s)

helm uninstall tenant-abc -n tenant-abc then kubectl delete ns tenant-abc — removes pods, services, ingress, secrets and PVCs. One namespace delete = total K8s teardown.

▼

Revoke Vault objects

Delete the AppRole datark-abc, its policy, the KV path secret/datark/tenants/abc/*, and (unless retaining ciphertext) the transit key datark-abc. Leaving these behind is a credential-hygiene leak. Note: deleting the transit key makes any retained tenant ciphertext undecryptable — gate behind the data-grace step.

▼

Confirm gone

Assert namespace absent, subdomain no longer resolves to a backend, and Vault AppRole/policy/KV removed. Emit deprovision receipt.

Because all three come from the same per-tenant Vault KV path, they line up by construction — that's the main reason to make Vault the single source.

8. Vault configuration & secrets model

This environment has its own dedicated Vault cluster (${VAULT_ADDR}) that you provision alongside the K8s cluster — separate from the production Koneksi Vault, which lives in a different NHN account and is never referenced here. It is the single source of truth for secrets. The provision script drives Vault with a provisioner token/AppRole, then materializes a per-tenant K8s Secret the pods consume via envFrom. (The transit/AppRole/KV pattern mirrors how the Koneksi backend already uses Vault, so the backend image works unchanged — only pointed at this env's Vault.)

What you provide for this environment (two things):
  1. A Kubernetes cluster — kube-admin connection. Hosts all tenant namespaces + shared ingress.
  2. A Vault cluster — reachable from the K8s cluster, with transit, approle, and KV v2 engines enabled, plus a provisioner token/AppRole for the script. Everything else (per-tenant AppRoles, policies, transit keys, KV secrets) the script creates.

Per-tenant Vault objects (created once, idempotently)

ObjectPath / namePurpose
KV v2 secretssecret/datark/tenants/<id>/*kripfs_cluster_secret, kripfs_static_bearer, mongo_pass, redis_pass, postgres_pass, app_key, jwt_secret
Transit keytransit/keys/datark-<id>Per-tenant PII KEK (isolation vs shared pii-kek)
Policydatark-tenant-<id>read that KV path + encrypt/decrypt on that transit key only

Shared platform secrets

Spaces / PortOne / Postmark / provenance creds live once at secret/datark/shared/* and are injected into every tenant backend — same value, read via the same provisioner step. (Postgres, mongo and redis creds are per-tenant, not here.)

Delivery model & rules

  • Chosen (now): script reads Vault → writes the per-tenant K8s Secret. Simplest, mirrors the existing envFrom secretRef pattern. Secret lands in etcd — acceptable to start.
  • Generate-once / stable: on first provision the script generates missing per-tenant values (openssl rand) and writes them to Vault KV; on re-run it reads existing values. Rotating cluster_secret or the transit key would break peering / make PII undecryptable — never regenerate blindly.
  • Least privilege: each tenant's AppRole can touch only its own KV path + transit key (policy-scoped). No tenant can read another's secrets.
  • Hardening path: swap the "materialize K8s Secret" step for the Vault Agent Injector or External Secrets Operator so secrets never persist in etcd. Design keeps this as a drop-in later change.
  • kripfs native Vault: the kipfs config has vault_addr/vault_token/vault_path fields — a later option to have kripfs pull its own secrets directly instead of via the K8s Secret.
  • Never committed. No tenant secret is written to git; the provisioner token is supplied at run time (env/Vault), never baked into the script.

10. Error handling & safety

  • set -euo pipefail + a rollback trap on ERR so failures never leave a half-built tenant.
  • Idempotent by construction — helm upgrade --install + lookup-guarded secrets mean re-running is safe.
  • Pre-flight gate refuses to start if a blocker (StorageClass, ingress, pull secret) is missing — a clear error beats a stuck deploy.
  • Quotas + NetworkPolicy cap blast radius: one tenant can't starve the cluster or reach another's pods.
  • Verification is the definition of done — provisioning only "succeeds" after the replication proof (§5).

11. Out of scope (this pass)

  • The marketplace storefront, payment, and the webhook that calls datark-provision.sh. This design gives that button a clean, single command to call.
  • Cluster-per-tenant / brand-new-cluster provisioning (needs cloud-provider APIs, not just a kube connection).
  • Cross-cluster kripfs federation and tenant data migration.
  • Autoscaling tiers up/down after purchase beyond a manual re-run with a new tier.
