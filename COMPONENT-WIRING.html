DatArk — Component Wiring & Connectivity

How every component connects to the others via environment variables, where each value comes from, and the upgrade/downgrade rules. Per tenant, on the Kubernetes cluster. 2026-07-15.

Legend — where each env value comes from
DNS in-namespace service address (non-secret)   CFG ConfigMap (non-secret)   SECRET K8s Secret (materialized from Vault)   VAULT Vault (source of truth / runtime transit)

Rule of thumb: addresses come from ConfigMap/DNS, credentials come from the Secret (which came from Vault). Every component reads its wiring from env — nothing is hard-coded.

3. The matching-value contracts (or the tenant breaks)

Because a few values must be identical across components, they all come from the same per-tenant Vault path (secret/datark/tenants/<id>) and are written to both keys — so they can never drift:

Vault valueUsed asBy
kripfs_static_bearerkoneksi_static_bearer · IPFS_AUTHORIZATION · KRIPFSDB_KIPFS_TOKENkipfs · backend · kripfs-db
postgres_passPOSTGRES_PASSWORD · KRIPFSDB_DB_PASSWORDpostgres · backend · kripfs-db
mongo_passMONGO_INITDB_ROOT_PASSWORD · in MONGO_CONNECTION_STRINGmongodb · backend
redis_passrequirepass · REDIS_PASSWORDredis · backend
kripfs_cluster_secretcluster_secretall kipfs pods

If any pair diverged: backend↔kipfs → 401 on every upload; backend/kripfs-db↔DB → auth failure; kipfs peers → cluster won't trust each other.

4. Upgrade & downgrade rules

Tiers scale kripfs node count, replication, and storage. What can change later — and in which direction:

AttributeUpgrade (↑)Downgrade (↓)Notes
CPU / memory per pod✅✅Free both ways — change limits/requests and re-apply
Namespace quota✅✅Both ways
kripfs node count (1↔3↔5)✅⚠️ riskyScale up adds pods. New uploads replicate; existing pins are not auto-copied to new pods (still readable via peering; a repin creates physical replicas). Scaling down needs a repin onto surviving nodes first or you lose replicas; removed pods leave orphaned PVCs.
Storage size (kripfs/mongo/postgres disk)⚠️ manual❌ neverBlock volumes only grow. Growing = a manual PVC-expand (a StatefulSet template change errors — immutable). Shrinking is impossible in place — only via data migration to a new smaller volume.
Key takeaway: treat tier storage as a floor you can raise but not lower. Compute and node count are flexible; disk is a one-way commitment. The provisioning automation supports upgrades only; downgrades (and storage growth) are manual operations, not a tier flip. Size lower tiers conservatively when pricing.
DatArk component wiring · addresses from ConfigMap/DNS, credentials from Secret (← Vault) · matching values come from one per-tenant Vault path so they never drift · storage upgrades are one-way.
