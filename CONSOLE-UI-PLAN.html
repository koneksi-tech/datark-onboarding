DatArk Console — Ops UI Plan & Mockups

A web console to monitor every tenant namespace, view exposed endpoints, and onboard a new tenant with one click (client name + tier). Plan, sample UI, and implementation. 2026-07-15.

What we're building
  • A web console (“DatArk Console”) for the ops team — runs inside the cluster with read access to all tenant-* namespaces.
  • Monitor: live deployment status per namespace (pods ready, workload health) + the exposed ingress endpoint per tenant.
  • Onboard: an Onboard button → enter client name + pick a tier → provisions a full isolated tenant (reusing the automation we already built).
  • Manage: per-tenant detail, upgrade tier, deprovision.

1. Architecture

A single small app (frontend SPA + a thin API) deployed in its own datark-console namespace. The API holds the Kubernetes access (via an in-cluster ServiceAccount + RBAC) — the browser never touches kube credentials. Provisioning reuses the existing pipeline.

3. How the Onboard button provisions (the key decision)

OptionHowPros / cons
A · In-cluster Job ★API creates a K8s Job from the provisioner image (helm+vault+scripts) that runs datark-provision.sh --tenant X --tier Y. API streams the Job logs to the UI.+ self-contained, fast, no external dep · + live logs · − must package the provisioner as an image + mount Vault/NCR/TLS secrets
B · GitHub ActionsAPI calls the existing provision-tenant.yml workflow_dispatch via the GitHub API; polls run status.+ reuses the proven, already-built pipeline · + zero new provisioning code · − needs a GitHub token, slower, logs live on GitHub
C · Operator/CRDA Tenant custom resource; a controller reconciles it. UI just creates the CR.+ most “Kubernetes-native”, declarative · − biggest build; overkill for now
Recommendation: Start with A (in-cluster Job) — it’s self-contained, gives live provisioning logs in the UI, and packages the automation we already wrote. Keep B as a fallback (the workflow already exists). Graduate to C only if this grows into a real product surface.

5. Implementation plan (phases)

1
Read-only dashboard — Console API with a read-scoped ServiceAccount; list tenant-* namespaces, pod/workload status, and ingress hosts. Frontend dashboard + tenant detail. (Highest value, lowest risk — pure read.)
2
Onboard action — add the provisioner Job (Option A): package bin/+chart as an image, API creates the Job with Vault/NCR/TLS secrets mounted, streams logs to the modal.
3
Manage actions — upgrade tier (re-run provision) and deprovision (run teardown). Guardrails: confirm dialogs, and the storage-is-one-way rule from the wiring doc.
4
Auth + polish — OIDC login, RBAC per ops role, audit log of who onboarded/deprovisioned, live pod status via websockets/watch.

6. Security & RBAC

  • Least privilege: the API ServiceAccount gets a ClusterRole to get/list/watch namespaces, pods, deployments, statefulsets, services, ingresses — read-only for monitoring. A separate, narrower Role grants create/delete for provisioning (namespaces + the provisioner Job).
  • No kube creds in the browser — the SPA only ever calls the Console API.
  • Secrets stay server-side — the UI shows status/endpoints, never secret values. Provisioning secrets (Vault token, NCR, TLS) are mounted only into the provisioner Job, not the API.
  • Auth-gate the console — internal ops tool; must sit behind login before it can onboard/deprovision.
  • Audit — log every onboard/upgrade/deprovision with the user + params.
