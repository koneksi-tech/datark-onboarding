1 · Symptom

Desktop app DatArkPOC/1.2.5 (Electron): login succeeds, but every file upload fails with “network error.” The web UI uploads to the same backend fine. So it was specific to the desktop upload path, not auth or the backend in general.

2 · How the desktop upload actually works

Unlike the web UI (which streams bytes through the backend), the desktop uses a direct-to-storage path. The backend only brokers the handshake; the file bytes go straight from the app to the kipfs cluster.

Key facts (from source):
  • Backend builds upload_url = IPFS_NODE_URL + "/cluster/add" and hands back the user’s own JWT as upload_authorization (init_upload_controller.go:222).
  • The desktop then POSTs the bytes directly to that upload_url on kipfs — this never touches the backend, which is why the backend logs showed init 200 but no “complete”.
  • kipfs receives a token that is not its static bearer, so it must call back to a Koneksi backend at POST /auth/validate to check it.

4 · Real root cause — the 401

The ingress-nginx access log showed the desktop did reach kipfs and POST /cluster/add — and got 401 “Koneksi auth failed.”

192.168.0.62  "POST /cluster/add HTTP/2.0" 401 ... "DatArkPOC/1.2.5 ... Electron/35.7.5"

Why kipfs said 401

kipfs validates a non-static token by calling POST {backend}/auth/validate. Which backend it uses has a fixed precedence (kipfs-net/src/koneksi_auth.rs):

  1. The Environment request header wins — production→api.koneksi.co.kr, uat→uat…, staging→staging….
  2. Only if that header is absent or unrecognized does it fall back to the configured koneksi_backend_url.
  3. If neither yields a URL → fail-closed → 401.

The demo kipfs had no koneksi_backend_url set. The desktop sends Environment: local (see below), which kipfs doesn’t recognize → no env URL → no fallback URL → fail-closed 401. The demo user’s JWT was never even checked against the demo backend.

Why the desktop sends Environment: local: the desktop bundles the latest released koneksi-engine (download-engine.ts pulls releases/latest). In that engine, DetectEnvironment("demo.koneksi.co.kr") matches none of uat/staging/api → hits the default → returns local. (The feat/detect-poc-domains engine change is not released, so it isn’t in the build.)

7 · Why no image rebuild was needed

  • kipfs: announce_addr and koneksi_backend_url are config fields the running binary already reads at startup (regenerated by the init container, applied on pod restart).
  • backend: already constructs upload_url from IPFS_NODE_URL — only the env value changed.
  • desktop: its Environment: local is exactly the value that triggers the koneksi_backend_url fallback — no engine/app change required.

8 · Remaining items & caveats

a) Persist in the Helm chart. announce_addr is already wired into the chart (value kripfs.announceAddr, default ""→null). koneksi_backend_url (#5) is currently live-patched only — it should be folded into the chart the same way so future tenants get it automatically and a helm upgrade doesn’t wipe it.
b) Large files → 502. One earlier attempt (a 64 MB file) got Connection reset by peer from kipfs. Small files are confirmed green; if large files still 502, that’s a separate kipfs body-size / timeout tune, not an auth issue.
c) Do NOT let the engine send Environment: production for tenants. The whole fix relies on the header being unrecognized (so kipfs uses koneksi_backend_url). If feat/detect-poc-domains (PR #145) ships and maps a tenant to production/uat/staging, kipfs routes back to the shared prod backend → 401 again. DatArk tenants must send a non-(production/uat/staging) value.
Bottom line: the current desktop build works against the demo tenant with the server-side config in place. Recommended next: (1) fold #5 into the chart, (2) confirm large-file behavior, (3) reconcile PR #145 with caveat (c).
