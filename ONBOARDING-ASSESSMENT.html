TECHNICAL ASSESSMENT

Why DatArk onboarding is not a "single image", and why Kubernetes is the right fit

Assessment of the "1-click, single-image, instantly available" expectation vs. the product's real component architecture. 2026-07-13.

Executive summary
  • "1-click" is delivered. A client (or the marketplace) triggers one action and gets a running, isolated DatArk environment. That part is done.
  • But "1-click" ≠ "one Docker image". DatArk is not one application — it is a distributed system of 5+ interdependent services, some of them stateful and multi-node. One image cannot represent it without breaking the product.
  • The single click triggers an orchestration (provision namespace → secrets → deploy 5 components → wire them → expose). The complexity is real; we hid it behind the click, not removed it.
  • Kubernetes is the most efficient fit for this architecture — the product's requirements map almost 1:1 onto native Kubernetes primitives. Alternatives (single container / VMs / compose) each break a core requirement.

1. The expectation vs. the product

The ask: "Sell it on the marketplace — the client clicks once, we deploy a single image, and the product is instantly available."

The reality: that mental model works for a single stateless app (one website, one API). DatArk is not that. It is a backup / decentralized-storage platform made of multiple cooperating services, several of which hold data and form a cluster. "One image" and "distributed storage cluster" are contradictory by definition.

2. Why it can't be a single image (the hard reasons)

  1. A container should run one process well. Cramming backend + 3 databases + a storage cluster into one image is the classic "fat container" anti-pattern: they can't scale, restart, or fail independently — one crash takes down everything, and you lose the ability to run multiple kripfs nodes at all.
  2. kripfs is a cluster, not a process. The product's core value is replication across nodes. That requires 3–5 separate, peered nodes with stable identities and addresses. A single image fundamentally cannot provide multi-node replication — the feature you are selling.
  3. State must persist. kripfs blocks, mongo, and postgres data must survive restarts and rescheduling. A single throwaway container loses all data on restart. You need persistent volumes attached to specific components.
  4. Every client must be isolated. A marketplace means tenant A cannot see tenant B's data, secrets, or storage. That needs isolation boundaries (namespace, network policy, per-tenant secrets, quotas) — not achievable inside one shared image.
  5. Components must find and authenticate to each other. backend→kripfs (bearer token), backend→databases (passwords), backend→Vault (AppRole). This wiring must be generated per tenant from a secrets source. That is orchestration, not packaging.

The single image that does exist: we built and published the kripfs image (kipfs:v0.0.1), and the backend image already existed. Those are the product's own images. The rest (redis/mongo/postgres) are standard database images. What the product needs on top is an orchestrator to place, connect, isolate, and scale them per client.

3. Challenges if this were NOT on Kubernetes

The same product on VMs (the earlier PoC model), bare metal, or docker-compose runs into these — each maps to something Kubernetes gives for free:

RequirementWithout Kubernetes (VMs / compose / bare metal)
Per-tenant isolationProvision separate VMs per client (minutes each, high cost) or hand-roll isolation. No cheap boundary like a namespace.
kripfs multi-node peeringManually wire static peer lists; adding a node means editing every node's config and restarting (the Ansible/VM approach). Error-prone at scale.
Self-healingA crashed process stays dead — you must build systemd watchdogs / custom supervision per component.
Scaling a tier (1→3→5 nodes)Provision new VMs, install, re-peer, re-balance — a manual multi-step operation per upgrade.
Persistent storageManually create/attach/track block volumes per component per tenant.
Secrets / TLS / networkingWired by hand for every tenant; no per-tenant secret injection.
On-demand provisioning ("1-click")The click must call cloud APIs to build VMs (minutes, quota, cost) instead of creating a namespace (seconds). Much slower, much more to fail.
Density / costVMs per tenant waste huge capacity; you cannot pack many tenants onto shared hardware.
Rollout / rollbackManual, per-host, no atomic upgrade or one-command rollback.

docker-compose specifically: fine for single-host dev, but no multi-host, no real per-tenant isolation, no self-healing, and a single point of failure — not viable for a multi-tenant marketplace product.

4. Assessment — is Kubernetes the most efficient way?

Verdict: Yes. For a product with this component architecture — multiple stateful + stateless services, a multi-node storage cluster, per-client isolation, on-demand provisioning, and elastic scaling — Kubernetes is the most efficient and appropriate platform. The product's requirements map almost one-to-one onto native Kubernetes primitives, so we implement each requirement by using the platform rather than building it.

Honest trade-offs

  • Kubernetes has a fixed setup cost (the cluster, Vault, ingress, storage). But it is paid once and amortized across every tenant — build the platform once, then each new client is seconds and near-zero marginal effort.
  • If DatArk were a single stateless app, a plain container on a PaaS (App Platform / Cloud Run) would be simpler and Kubernetes would be overkill. It is not that — it is a stateful, clustered, multi-tenant system, which is exactly Kubernetes' sweet spot.
  • Possible simplification (future): some datastores could be shared with per-tenant databases (one postgres/mongo cluster, a DB per client) instead of one per tenant, to raise density. kripfs must stay per-tenant and multi-node — that is the product's core.

5. So how is it still "1-click"?

We did not remove the complexity — we put it behind a single trigger. That is exactly how every SaaS marketplace product works: nobody ships a multi-service platform as one image; they ship an orchestrated deployment that a single click kicks off.

6. Recommendation

  • Keep Kubernetes. It is the efficient, correct platform for DatArk's architecture; the alternatives break a core product requirement (isolation, multi-node storage, self-healing, or fast provisioning).
  • Frame "1-click" honestly to stakeholders: one click to trigger, an orchestrated deploy underneath — already delivered (proven via the manual-trigger workflow).
  • Next efficiency levers (optional): shared databases with per-tenant DBs to raise density; Vault HA; and connecting the marketplace "buy" event directly to the existing provisioning workflow.
