1. Overview & purpose

DatArk is sold in a marketplace. When a client subscribes, clicking Onboard must provision a complete, fully isolated DatArk stack for that client — with no manual ops. Each tenant gets its own decentralized storage (kripfs), application backend, datastores (MongoDB, PostgreSQL, Redis), and a database-backup daemon (kripfs-db) — all in a dedicated Kubernetes namespace, sized by subscription tier.

The two things you provide once are a Kubernetes cluster and a Vault cluster. Everything per-tenant — namespace, secrets, TLS, DNS routing, encryption keys, the admin account — is generated automatically by the provisioning pipeline.

Current state: the full pipeline is built and validated end-to-end — a freshly provisioned tenant round-trips a file upload & download through the backend→kripfs path byte-for-byte, with zero manual fixes. An ops console at console.koneksi.co.kr onboards, monitors, upgrades, and tears down tenants.

2. Why Kubernetes, not a single image

The intuitive expectation is "one Docker image = one click." It cannot be a single image because a tenant is six cooperating services with independent lifecycles, storage, and scaling:

  • Stateful, independently-scaled parts. kripfs is a clustered storage system that scales from 1 to 5 peered nodes with per-node persistent volumes; the databases each need their own durable volume. A single container can't hold multiple independently-scaled StatefulSets with separate disks.
  • Isolation is a hard requirement. Each client's data, credentials, and encryption keys must be separated. Kubernetes namespaces + NetworkPolicies + per-tenant Vault paths give real isolation on shared hardware; a single image on a shared host does not.
  • Independent failure & upgrade. Restarting the backend must not restart storage; growing storage must not touch the databases. Separate workloads make this safe.

Kubernetes is the efficient fit: one shared cluster runs many tenants at high density, the same Helm chart renders every tenant, and "1-click" is delivered by automation on top of Kubernetes (a console button → pipeline → Helm), not by collapsing the product into one container.

7.1 Upgrade & downgrade rules

An upgrade is a one-click tier flip (console) that re-runs the pipeline at the higher tier and only ever grows kripfs. Kubernetes forbids resizing a StatefulSet's volume via Helm, so kripfs storage is grown out-of-band via online Cinder volume expansion; the databases never change size, keeping the upgrade off the immutable-volume path.

Attributefree→starter→pro→enterpriseUpgrade ↑Downgrade ↓How
kripfs node count1→2→3→5✅ automated❌ not supportedScale up adds pods; each gets a fresh volume and peers in, replicating existing CIDs. Existing pods/volumes untouched.
kripfs replication factor1→1→2→3✅ automated❌Config, applied on the rolling update — more physical copies per CID.
kripfs storage / pod20→50→100→250Gi✅ automated (grow-in-place)❌ neverPipeline raises the quota, orphan-deletes the STS (pods/PVCs stay bound — no downtime), online-expands the PVCs, then Helm recreates the STS at the new size.
kripfs CPU / memoryscales per tier✅✅Pod template — rolling update; data on the PV is preserved.
Namespace quotascales per tier✅ automated⚠️Raised before expansion. Lowering blocked if usage exceeds the new limit.
mongo / postgres storage20Gi — fixed, all tiers——Never changes — this is what keeps helm upgrade off the immutable-volume path.
Data safety: capacity grows by adding pods and by expanding volumes in place — the underlying physical disks are preserved. Downgrade is intentionally one-way / unsupported: block storage can't shrink and removing kripfs peers can drop a CID's last replica. Verified: free→pro grew kripfs 1→3 nodes and 20→100Gi on the same volume, databases untouched.

8. Component wiring & the matching-value contracts

Addresses come from the ConfigMap / in-namespace DNS; credentials from the Secret (materialized from Vault). Several values must match exactly across components or the tenant silently breaks — these are now generated from a single Vault source and verified at provision time.

10. Provisioning — how to onboard

10.1 Three ways to trigger

  • Ops console (primary): open console.koneksi.co.kr → + Onboard Tenant → enter client name + tier → confirm. The console dispatches the pipeline.
  • GitHub Actions: workflow provision-tenant.yml → Run workflow → client name + tier.
  • Script (operator): ./bin/datark-provision.sh --tenant <name> --tier <free|starter|pro|enterprise>

10.2 What the automation does (idempotent, with rollback on a failed fresh create)

  1. Create namespace tenant-<client>; label it with tenant + tier; create the ncr-regcred pull secret.
  2. Sync the wildcard TLS secret into the namespace (Ingress needs it namespaced).
  3. Vault: create the transit key, generate-once KV secrets (incl. admin creds), write the policy (with datakey/plaintext), create the AppRole.
  4. Materialize the Kubernetes Secret from Vault (incl. the Bearer-prefixed IPFS_AUTHORIZATION).
  5. Contract guardrail — fail loudly if the Vault policy lacks datakey or IPFS_AUTHORIZATION lacks the Bearer prefix.
  6. (Upgrade only) grow kripfs storage via online PVC expansion.
  7. helm upgrade --install the datark-tenant chart with the tier values.
  8. Seed the tenant admin (register via API → promote in Mongo).
  9. Emit a JSON summary (URL, admin email, Vault refs).

10.3 Teardown

Delete the namespace (console Destroy or kubectl delete ns tenant-<client>) — removes all workloads, PVCs, and secrets. Vault objects are retained by default (the transit key is kept so retained ciphertext stays decryptable; set DELETE_TRANSIT=true to crypto-shred).

11. Admin & developer handover

The backend gates new end-user signups behind admin approval + email verification but seeds no admin. So provisioning now bootstraps one admin per tenant automatically: it generates admin@<client>.koneksi.co.kr + a strong password (generate-once in Vault), registers it via the API, then promotes it in Mongo (approve + verify + assign the system_admin role).

Per-tenant handover to the developer is exactly two things:
  • API base URL — https://<client>.koneksi.co.kr (the same API the Koneksi web app already targets; integrate the web UI against it).
  • Admin login — admin@<client>.koneksi.co.kr + password (shown in the console tenant detail, and stored at secret/datark/tenants/<client> key admin_password). The developer uses it to approve their own end-users.
The developer never touches Vault, kripfs, or the bearer — those are internal and automated.

API surface: end-user flow — POST /users/register → admin approves → POST /tokens/request (email+password → JWT) → POST /files/upload / GET /files/:id/download. Programmatic flow — /clients/v1/… with an API key (scopes user_file:upload/download). Admin flow — /admin/* (e.g. GET /admin/users/list).

12. Reliability — bugs fixed & guardrails

Validating the full upload/download path surfaced three provisioning bugs that would have broken every tenant. All are fixed, and a guardrail prevents the first two from silently regressing.

#BugSymptomFix
1Vault policy missing datakey/plaintextregister 403 "failed to create profile" (backend envelope-encrypts PII)Added datakey/plaintext to the tenant policy
2IPFS_AUTHORIZATION without Bearer prefixupload 401 "Koneksi auth failed" from kripfsSet it to Bearer <token>
3Backend boots before its datastores (startup race)"Session revoked: logged in from another device" → uploads blocked (Redis disabled at startup)wait-for-datastores initContainer on the backend
Guardrail: assert_tenant_contract fails provisioning loudly if the Vault datakey grant or the Bearer prefix ever regress. Validated clean-room: destroy + fresh provision + register/login/upload/download round-trips byte-for-byte with zero manual fixes.

Note: tenants provisioned before these fixes need a re-provision (idempotent) to be repaired.
