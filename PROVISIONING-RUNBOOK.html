DatArk Provisioning — Process Flow

How to provision a new client (tenant) on the datark-cluster using the GitHub Action. One manual click → a fully isolated DatArk stack.

TL;DR
  • Repo: koneksi-tech/datark-onboarding → Actions tab → Provision DatArk tenant → Run workflow.
  • Enter client name + pick tier → the automation builds the whole stack in namespace tenant-<client>.
  • Result: a private DatArk env reachable at https://<client>.koneksi.co.kr (after a DNS record).

A. Operator steps — how to trigger

Open the workflow

GitHub → koneksi-tech/datark-onboarding → Actions tab → left sidebar Provision DatArk tenant.

▼

Run workflow

Click Run workflow (top-right). A form appears.

▼

Fill the inputs

client_name — the client's name. Becomes the tenant id + namespace tenant-<name> and URL <name>.koneksi.co.kr. Use lowercase letters/numbers/hyphens (e.g. acme).

tier — dropdown: tier1 / tier2 / tier3 (see mapping below).

▼

Start + watch

Click the green Run workflow. Open the run to watch live logs. On success, the Summary shows the tenant URL.

B. What the automation does (under the hood)

The workflow runner executes these steps automatically after you click Run:

Prepare runner

Checks out the repo, maps tier1/2/3 → starter/pro/enterprise, installs kubectl + helm + vault.

▼

Load credentials (from repo secrets)

Writes the kubeconfig, NCR pull secret, and *.koneksi.co.kr TLS cert/key from GitHub Secrets; exports Vault address + provisioner token.

▼

Validate + pre-flight

Checks the client name/tier, and that the cluster, StorageClass, ingress and Vault are all reachable. Fails fast if not.

▼

Namespace + pull secret + TLS

Creates namespace tenant-<client>, the ncr-regcred image-pull secret, and the wildcard TLS secret.

▼

Vault (per-tenant)

Creates a transit key, generates per-tenant secrets (kripfs bearer/cluster-secret, mongo/redis/postgres passwords, app keys), and a scoped policy + AppRole — all under secret/datark/tenants/<client>.

▼

Materialize K8s Secret

Reads the tenant's secrets from Vault (plus shared platform secrets) → writes the in-namespace <client>-secret the pods consume.

▼

Deploy the stack (Helm)

Installs kripfs (N nodes) + backend + redis + mongodb + postgres + ingress + NetworkPolicy + ResourceQuota for the chosen tier.

▼

Output

Prints the result in the run Summary: namespace, tier, and https://<client>.koneksi.co.kr.

C. After provisioning — make it reachable

  1. Add DNS — one A record in the koneksi.co.kr zone: 
    <client>.koneksi.co.kr.   A   133.186.214.160
    (133.186.214.160 = the shared ingress LoadBalancer.)
  2. Access — https://<client>.koneksi.co.kr — valid TLS (covered by the *.koneksi.co.kr cert).
  3. Verify without DNS (optional): 
    curl --resolve <client>.koneksi.co.kr:443:133.186.214.160 \
     https://<client>.koneksi.co.kr/check-health

D. Isolation — what each client gets

  • Own namespace tenant-<client> — pods, storage, secrets private to it (NetworkPolicy + ResourceQuota).
  • Own kripfs cluster (1/3/5 nodes by tier) with its own bearer token — tenant A cannot read tenant B's storage.
  • Own databases (mongo + postgres + redis) with per-tenant passwords from Vault.
  • Own Vault path + transit key — secrets and encryption scoped per tenant.

E. Upgrade & teardown

ActionHow
Upgrade tier (more kripfs nodes)Re-run the workflow for the same client with a higher tier. Data is retained (PVCs kept). Note: a tier whose DB storage size differs can hit an immutable-volume error — grow storage manually if needed.
DeprovisionCLI: ./bin/datark-deprovision.sh --tenant <client> --force — deletes the namespace (pods, PVCs, secrets) and the tenant's Vault objects.
Prerequisites (one-time, already done):
  • Repo secrets set: KUBE_CONFIG, VAULT_ADDR, VAULT_TOKEN, NCR_DOCKERCONFIG, TLS_FULLCHAIN, TLS_KEY.
  • Cluster ready: StorageClass, ingress-nginx + LB 133.186.214.160. Vault ready at its address. kripfs image kipfs:v0.0.1 in NCR.
